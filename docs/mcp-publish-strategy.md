# NTLSN × MCP — publishing sector data to AI agents

**Status:** strategy / research baseline (2026-07-10). Feeds into [Increment 1 plan](./mcp-increment-1.md).
**Author of artefact:** Claude (deep-research run `wf_119ddbd6-4b4`, 96/101 agents, 23/25 claims verified 3-0 against primary sources).

## Bottom line

Extend the **existing** Netlify-Functions MCP server into a remote, read-only endpoint that publishes NTLSN's sector data (events, archive, universities, scholarly look-ups) so AI agents — Claude Desktop, Claude Code, other MCP clients — can discover, query and consume it. Ship it **unauthenticated first**, **lead with tools**, return **Schema.org `Event` JSON-LD** from the event tools, and reuse proven **cost-control patterns** so external look-ups can't run up the Netlify bill.

## The one constraint that shapes everything

> **Claude's `mcp_servers` connector only consumes *tools*.** Resources, prompts and resource templates are spec-correct for publishing read-only data, but they are **not reachable through Claude's connector today** (you would need to run your own MCP client via the SDK). So although "resources" sounds like the right primitive, **tools-first is the pragmatic path to being consumable by Claude now.** Add resources/templates in parallel for future-proofing and richer clients. *(Verified 3-0, Anthropic connector docs — "Of the feature set of the MCP specification, only tool calls are currently supported.")*

## Current state of the NTLSN MCP server (grounded, not assumed)

The server already exists and is in production at `https://www.ntlsn.com/.netlify/functions/mcp`:

- **3 tools** already defined: `upcoming_events`, `search_archive` (1,431 ALTC/OLT-funded works), `universities`.
- **Plain JSON-RPC 2.0 over HTTP POST** + a GET info endpoint. `protocolVersion: "2024-11-05"`.
- **Data origin:** reads the CORS-open `/data/*.json` from its own deploy URL (`DEPLOY_PRIME_URL || URL`), with a 60s module-scope cache and an 8s fetch timeout. Clean pattern — new tools reuse it for free.
- **Schema.org `Event` JSON-LD already exists** as `events-ld.json` (generated by `scripts/build-feeds.mjs`, embedded in the page as `ntlsn-events-ld`). The "one payload for humans + agents" angle is therefore partly built — a tool can return the same shape.
- **Transport caveat (the make-or-break item):** the current endpoint is a simplified JSON-RPC-over-POST at the *original* protocol version. Remote MCP as the Claude connector expects it is **Streamable HTTP** (content negotiation, `Mcp-Session-Id`, `Accept: application/json, text/event-stream`). Whether the current plain-POST endpoint is consumed by the connector as-is **must be verified first** — see Increment 1, Step 0.

## Landscape (verified findings)

### 1. Transport — settled direction, verify the current server against it
Remote MCP runs **only** over the **Streamable HTTP** transport. `stdio` is local/subprocess-only and cannot serve remote clients; the legacy HTTP+SSE transport was **deprecated (March 2025)**. **Netlify's own guide** implements MCP statelessly as transactional HTTP (`sessionIdGenerator` undefined — short request/response, not long-lived SSE), which is a drop-in fit for Netlify Functions. NTLSN's serverless topology is the *supported* pattern, not a workaround. *(3-0; MCP spec + Netlify guide + ORCID remote server on protocol 2025-11-25.)*

### 2. Auth — optional, so phase it
MCP authorization is **OPTIONAL**. HTTP servers *should* conform to **OAuth 2.1** when they authenticate, acting as a **resource server** accepting **Bearer** tokens (the auth server can be co-hosted or separate). A **public read-only server legitimately needs zero auth** — `context7` is cited as a live example. So NTLSN ships public now and adds OAuth only when something non-public is gated. *(3-0; MCP Authorization spec + connector docs.)*

> ⚠️ The *granular* 2026-draft OAuth detail (PKCE mandates, RFC 9728/8414/8707, Client ID Metadata Documents, the reported Dynamic-Client-Registration deprecation) **could not be fully verified** — 2 of 3 verifiers hit a rate limit. Treat the high-level "OAuth 2.1 for HTTP transports" as solid, but do **not** commit to an auth build on the granular RFC detail until re-checked.

### 3. Tools vs resources vs templates
- **Tools** — model-controlled; what the AI invokes. **This is the consumption path today.**
- **Resources / resource templates** — application-controlled, URI-addressed, paginated (`cursor`/`nextCursor`); templates give parameterized per-entity URIs like `event://{id}` or `person://{orcid}` without pre-registering each one. The spec *recommends a custom/non-https scheme* (e.g. `event://`), which favours NTLSN's design.
- **Caveat:** pagination/template support is **uneven across clients** (reported `nextCursor` bugs in `claude-code` and `openai/codex`). Reinforces tools-first as the reliability hedge. *(3-0; MCP Resources spec + FastMCP.)*

### 4. Schema.org `Event` / JSON-LD — one payload for humans + agents
Schema.org defines a stable **`Event`** type (`startDate`/`endDate`, `location`, `organizer`, `performer`, `eventAttendanceMode`, `eventStatus`, `offers`, `subEvent`/`superEvent`, `eventSchedule`). It **natively models streamed/hybrid symposia**: `eventAttendanceMode: MixedEventAttendanceMode` + a `VirtualLocation` carrying the stream URL. NTLSN already emits this for the web page (`events-ld.json`), so a tool can return the identical shape. *(Minor caveat: the virtual/hybrid vocabulary is still in schema.org's "new" area, but it is Google-recognised and adopted on 100K–1M domains.)* *(3-0; schema.org/Event.)*

**Honest gap for NTLSN:** `data/events.json` has no per-event stream URL and no speakers — so the `VirtualLocation` and `performer` properties can't be populated from current data. That mapping waits on a data addition, not on MCP work.

### 5. Scholarly-gateway pattern — proven precedent + free cost controls
Community MCP servers already do exactly what NTLSN wants, over Streamable HTTP, identifier-routed:
- **ORCID server** (`orcid.caseyjhand.com`, protocol `2025-11-25`) — dual tools + resource templates; returns DOIs/PMIDs/arXiv IDs *ready to chain* into Crossref/PubMed/arXiv.
- **`crossref-mcp-server`** — DOI resolution + search; stdio **and** Streamable HTTP, env-switchable; pluggable `none`/`jwt`/`oauth` auth.
- **`academic-tools-mcp`** — a 7-provider gateway with **code-verified cost controls directly reusable on Netlify**: per-provider concurrency semaphores, single-flight request coalescing, `Retry-After`-honouring retries, **burst backpressure** (the Nth+1 caller fails fast with `{retryable:true}` so the agent learns to slow down), and **24-hour negative caching of 404s** to stop retry-happy agents burning quota.
- **Zotero MCP** — ~40 tools, DOI/ISBN/URL lookups and open-access PDF cascading.

> The cost-control patterns are the single biggest lever for the "Netlify is costing me too much" concern — they cap per-invocation spend on external look-ups **without any edge/CDN product.**

*(3-0, the cost-control claim verified by source-code inspection, not just docs.)*

## Recommended direction

1. **Verify then modernise transport** (Streamable HTTP, stateless) so the connector actually consumes the endpoint — see [Increment 1](./mcp-increment-1.md), Step 0.
2. **Add read-only tools** grounded in data NTLSN already has: `event_detail`, plus pass-through `scholarly_lookup` (ORCID/DOI/Crossref) and the under-used `rcf` / `best_practice` datasets.
3. **Return Schema.org `Event` JSON-LD** from the event tools (reuse `events-ld.json`'s shape).
4. **Borrow the cost controls** (negative-cache 404s, single-flight, per-tool concurrency cap).
5. **Defer** full OAuth, MCP-Registry listing, resource/templates, and any tool that needs data NTLSN doesn't yet hold (speakers, per-event stream URLs).

## Don't build yet
- **Full OAuth 2.1** — optional, and the granular RFC detail is unverified (above).
- **MCP Registry listing / `.well-known`** — premature until there is partner demand; the listing process and whether it requires OAuth is an open question.
- **Heavy resource-template investment** — wait for the connector to support resources.
- **`people` / `speakers` tool** — no data source exists yet.

## Gaps / honest unknowns
- **Biggest design lever, unresolved:** will/when will Claude's connector add resources + prompts? Determines whether to invest in resource templates now or stay tools-first.
- **Auth granularity** unverified (above).
- **Netlify cold-start + invocation cost at real agent-query volume** was not quantified by any verified source — watch the Functions meter once the tools are live.
- Most scholarly-MCP existence proofs (esp. Zotero) are **local/stdio/personal-library** — they prove feasibility + transport, not NTLSN's exact remote/public/serverless topology. The ORCID and Crossref servers are the closer models.
- Minor staleness: the `crossref-mcp-server` README's "~155M works" is stale — Crossref is currently ~**176–180M** records.
